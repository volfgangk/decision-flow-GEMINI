# decision-flow-claude
🤖 Decision Flow - Claude Version (A/B 비교 트랙 B)

# 비개발자용 바이브코딩 운영 체크리스트

## 0. 프로젝트 한 줄 정의
- [ ] 만들고 싶은 것을 한 문장으로 적었다.
- [ ] 이걸 왜 만드는지 적었다.
- [ ] 누가 쓸지 적었다.
- [ ] 성공 기준을 3개 이하로 정했다.
- [ ] 첫 버전에서 하지 않을 기능을 정했다.

## 1. 기획 고정
- [ ] PRD를 작성했다.
- [ ] 기능정의서를 작성했다.
- [ ] 화면 목록을 적었다.
- [ ] 데이터가 어디에 저장되는지 정했다.
- [ ] 입력/출력 형식을 적었다.
- [ ] 변경되면 안 되는 규칙을 따로 적었다.

## 2. 모델 의존성 낮추기
- [ ] 주력 모델 1개를 정했다.
- [ ] 백업 모델 1개를 정했다.
- [ ] Claude 전용 문장 대신 범용 프롬프트로 바꿨다.
- [ ] 중요한 문서는 AI 기억에 맡기지 않고 내 문서에 저장했다.
- [ ] 모델이 바뀌어도 유지될 규칙을 문서화했다.
- [ ] 서드파티 도구 연결을 꼭 필요한 것만 남겼다.

## 3. 추상화 레이어
- [ ] 모델 호출을 한 곳에서만 관리하도록 정했다.
- [ ] 프롬프트 템플릿을 폴더에 모았다.
- [ ] 출력 형식을 하나로 통일했다.
- [ ] 바뀌기 쉬운 설정값은 따로 분리했다.
- [ ] 같은 작업을 다른 모델로도 돌려볼 수 있게 준비했다.

## 4. 프롬프트 운영
- [ ] 한 번에 묻는 범위를 작게 쪼갰다.
- [ ] 분석, 설계, 구현 순서로 요청한다.
- [ ] 출력 형식을 항상 지정한다.
- [ ] 오류가 나면 전체 재작성보다 부분 수정부터 시킨다.
- [ ] 잘 된 프롬프트를 저장한다.
- [ ] 실패한 프롬프트도 저장한다.

## 5. 개발 진행
- [ ] 기능을 1개씩만 추가한다.
- [ ] 화면, 저장, 알림, 로그인처럼 큰 덩어리로 나눴다.
- [ ] 한 번에 여러 기능을 시키지 않는다.
- [ ] 기능이 생기면 바로 실행해 본다.
- [ ] 문제 생긴 부분을 메모한다.

## 6. 검증
- [ ] 버튼이 눌리는지 확인했다.
- [ ] 새로고침 후에도 데이터가 남는지 확인했다.
- [ ] 모바일 화면에서 확인했다.
- [ ] 빈 값 입력을 확인했다.
- [ ] 긴 값 입력을 확인했다.
- [ ] 특수문자 입력을 확인했다.
- [ ] 네트워크 오류 상황을 확인했다.
- [ ] 다른 사람이 써도 이해되는지 확인했다.

## 7. 버전 관리
- [ ] 버전 번호를 붙였다.
- [ ] 각 기능이 어느 버전에 들어갔는지 기록했다.
- [ ] 완료와 검수 완료를 분리했다.
- [ ] 변경 이유를 한 줄씩 적었다.
- [ ] 다음 버전으로 넘길 것을 분리했다.

## 8. 비용 관리
- [ ] 호출이 많은 작업은 묶어서 보냈다.
- [ ] 반복 수정은 저렴한 모델로 먼저 돌렸다.
- [ ] 중요한 작업은 한 모델에 몰빵하지 않았다.
- [ ] 외부 도구 연결을 최소화했다.
- [ ] 한 달에 한 번 유지 여부를 점검한다.

## 9. 문서화
- [ ] 요구사항 문서를 저장했다.
- [ ] 프롬프트를 저장했다.
- [ ] 잘 된 결과를 저장했다.
- [ ] 실패 사례를 저장했다.
- [ ] 변경 로그를 저장했다.
- [ ] 나중에 옮길 수 있게 정리했다.

## 10. 운영 루틴
- [ ] 오늘 할 일 3개만 정했다.
- [ ] 30분 단위로 결과를 확인했다.
- [ ] 기능이 되면 저장하고 기록했다.
- [ ] 막히면 질문 범위를 줄였다.
- [ ] 하루 끝에 다음 작업 1개만 남겼다.
